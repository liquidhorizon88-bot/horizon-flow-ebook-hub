Why return-window deadlines need source control

When a customer asks “Can I still return this?” the safe answer depends on which date starts the clock, what proof is required, whether the product is final sale, whether the item is defective, and which page or marketplace policy controls. If AI tools compress those details into a single generic deadline, the business can create unsupported refunds, chargebacks, review disputes, and support escalations.

STOP PUBLISHING: do not let AI publish, rewrite, or summarize return-window answers when purchase/delivery dates conflict, holiday extensions are unclear, proof-of-purchase is missing, condition rules differ by product, final-sale items are involved, or owner/support review is required.

Safe snippets to use before promising eligibility

Return-window review: Please send the order number and purchase or delivery date. We will check the current return policy, item condition, and any product-specific exceptions before confirming eligibility.
Source-control note: This return-window answer is not ready for AI/search publication until the start date, deadline, proof requirements, exceptions, and owner review date are documented.
Do-not-invent rule: Do not promise that an item is still returnable, that a grace period applies, that a holiday extension is active, or that proof is optional unless the verified source explicitly says so.

Weekly source-health questions

  • Which product, FAQ, checkout, email, or marketplace pages disagree about when the return window starts?
  • Which AI/chatbot answers imply a grace period or holiday extension without an active source?
  • Which downloadable PDFs, receipts, or order emails still mention an old return deadline?
  • Which product exceptions need owner/support review before they appear in answer-engine snippets?
